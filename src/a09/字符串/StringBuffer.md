# 🥊 StringBuffer vs StringBuilder：内存机制与终极对决

> **创建时间**：2026-03-09
> **标签**：#Java基础 #String #多线程 #性能优化

在 Java 中，当我们需要频繁拼接或修改字符串时，直接使用 `String` 会产生大量垃圾对象。为了解决这个问题，Java 提供了两个可变字符串类：`StringBuffer` 和 `StringBuilder`。

它们俩像是一对双胞胎兄弟：**长相（API 方法）和内在（内存机制）几乎一模一样，唯一的区别在于“性格（是否支持并发）”。**

---

## 一、 共同的内存机制 (底层数组与扩容)

无论是 `StringBuffer` 还是 `StringBuilder`，它们底层的内存运作机制是完全相同的。

### 1. 底层数据结构：可变数组
它们都不是像 `String` 那样的不可变对象。它们内部维护着一个**可以修改内容的字符数组**（在 JDK 9 之后是 `byte[]` 数组，为了节省空间）。

### 2. 初始容量 (Capacity)
当你 `new` 一个空对象时，它默认会在内存中创建一个长度为 **16** 的数组。
    
        StringBuffer sb = new StringBuffer();     // 默认容量 16
        StringBuilder sb2 = new StringBuilder();  // 默认容量 16
### 3. 扩容机制 (动态成长)
当你不断 append() 追加字符，直到这 16 个格子装不下时，它会在内存中进行“搬家”（扩容）：
- 默认扩容公式：新容量 = (旧容量 * 2) + 2。
- 搬家过程：在堆内存中开辟一块按公式算出的新数组空间，然后把老数组里的数据**复制（拷贝）**过去，最后抛弃老数组（等待 GC 回收）。
- 特殊情况：如果一次性追加了超长的字符串，公式算出来的新容量依然不够，它就会以实际需要的长度为准直接分配。
- 💡 性能忠告：因为扩容需要复制数组，非常消耗性能。如果你提前知道要拼接的字符串大概多长（比如 1000 个字符），请务必在创建时指定初始容量：new StringBuilder(1000);。
## 二、 核心区别：线程安全 vs 运行速度
既然内存机制一样，为什么还要造两个类？这是为了应对不同的工作场景（单线程 vs 多线程）。
1. StringBuffer：稳重老实（线程安全，但慢）
特点：它是线程安全的。
底层原理：你去翻看 StringBuffer 的源码，会发现它的绝大多数方法（如 append, insert, delete 等）都加上了 synchronized 关键字（同步锁）。
打个比方：它就像一个只有一个坑位的公共厕所。多个线程（人）同时想进去修改字符串，必须排队。一个人进去了，门就锁上（上锁），别人只能在外面等他出来（释放锁）才能进去。
缺点：频繁地加锁、释放锁，导致它的运行速度比较慢。
2. StringBuilder：奔放狂野（非线程安全，但极快）
特点：它是非线程安全的（也就是线程不安全的）。
底层原理：它在 JDK 1.5 才诞生。你去翻看它的源码，它的方法没有任何锁（没有 synchronized）。
打个比方：它就像一片野地。如果没有别人，你一个人怎么折腾都行，速度极快。但是，如果多个线程（人）同时冲进去修改同一个字符串，就会发生踩踏事件（数据错乱、丢失字符甚至报错）。
优点：因为它省去了排队和加锁的开销，它的运行速度极快！
